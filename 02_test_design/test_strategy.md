# 테스트 전략

## 1. 문서 목적

본 문서는 PC방 매장 운영 플랫폼 QA 포트폴리오의 테스트 전략을 정의한다.

본 프로젝트는 단순 기능 동작 확인이 아니라, 실제 매장 운영에 영향을 줄 수 있는 품질 리스크를 중심으로 테스트를 설계하는 것을 목표로 한다.

특히 사용자 주문, 관리자 주문 처리, 결제 실패, 쿠폰/포인트 복원, 정산 반영, 권한별 접근 제어와 같이 운영 안정성에 직접적인 영향을 주는 영역을 중점적으로 검증한다.

본 문서는 이후 테스트케이스 설계, AI 테스트케이스 생성, Playwright E2E/API 자동화 대상 선정의 기준으로 사용된다.

---

## 2. 테스트 대상 서비스 요약

PC방 매장 운영 플랫폼은 일반 사용자, 매장 직원, 매장 관리자, 본사 관리자가 사용하는 운영 관리 서비스이다.

| 사용자 유형 | 주요 역할                                             |
| ----------- | ----------------------------------------------------- |
| 일반 사용자 | 로그인 후 좌석을 선택하고 음식 주문 및 결제를 진행    |
| 매장 직원   | 접수된 주문을 확인하고 접수, 조리중, 완료 상태로 처리 |
| 매장 관리자 | 담당 매장의 주문, 좌석, 매출, 정산 현황 확인          |
| 본사 관리자 | 전체 매장 정산 조회 및 쿠폰 정책 관리                 |

본 포트폴리오의 핵심 검증 영역은 다음과 같다.

| 영역        | 주요 검증 내용                                   |
| ----------- | ------------------------------------------------ |
| 회원/로그인 | 로그인, 로그아웃, 권한별 메뉴 노출               |
| 좌석        | 좌석 선택, 사용 중 상태 변경, 중복 선택 방지     |
| 주문        | 음식 주문 생성, 관리자 화면 반영, 주문 상태 변경 |
| 결제        | 결제 성공, 결제 실패, 중복 결제 방지             |
| 쿠폰/포인트 | 쿠폰 사용, 포인트 사용, 취소 시 복원             |
| 정산        | 완료 주문 기준 매출 반영, 취소/실패 주문 제외    |
| 권한        | 사용자별 접근 제어, 매장 간 데이터 접근 제한     |

---

## 3. 테스트 목표

본 테스트의 주요 목표는 다음과 같다.

1. 사용자가 로그인 후 좌석 선택, 음식 주문, 결제까지 정상적으로 진행할 수 있는지 검증한다.
2. 사용자 주문이 매장 관리자 화면에 정상적으로 반영되는지 검증한다.
3. 주문 상태 변경 결과가 사용자 화면과 관리자 화면에 일관되게 표시되는지 검증한다.
4. 결제 실패 시 주문, 쿠폰, 포인트, 정산 데이터가 잘못 변경되지 않는지 검증한다.
5. 주문 취소 시 사용한 쿠폰과 포인트가 정상 복원되는지 검증한다.
6. 완료된 주문만 정산 금액에 반영되는지 검증한다.
7. 일반 사용자, 매장 직원, 매장 관리자, 본사 관리자 간 권한이 정확히 분리되는지 검증한다.
8. 매장 A 관리자가 매장 B 데이터에 접근할 수 없는지 검증한다.
9. 반복 검증이 필요한 핵심 회귀 영역을 Playwright 기반 자동화 대상으로 선정한다.

## 4. 핵심 품질 기준

본 프로젝트에서 중요하게 보는 품질 기준은 다음과 같다.

| 품질 기준        | 설명                                                               | 주요 검증 영역           |
| ---------------- | ------------------------------------------------------------------ | ------------------------ |
| 기능 정확성      | 요구사항대로 기능이 동작하는지 확인                                | 로그인, 좌석, 주문, 결제 |
| 데이터 정합성    | 주문, 결제, 쿠폰, 포인트, 정산 상태가 일관되는지 확인              | 결제, 쿠폰, 정산         |
| 권한 통제        | 권한 없는 사용자의 화면/API 접근이 차단되는지 확인                 | 사용자 권한, 매장 권한   |
| 예외 처리        | 결제 실패, 품절, 중복 클릭 등 예외 상황이 안전하게 처리되는지 확인 | 주문, 결제, 쿠폰         |
| 상태 전이 안정성 | 주문 상태가 정책에 맞는 순서로만 변경되는지 확인                   | 주문 상태 관리           |
| 회귀 안정성      | 배포 전 핵심 기능을 반복 검증할 수 있는지 확인                     | E2E/API 자동화           |

## 5. 테스트 우선순위 기준

테스트 우선순위는 비즈니스 영향도, 발생 가능성, 데이터 영향도, 자동화 필요성을 기준으로 High, Medium, Low로 구분한다.

| 우선순위 | 기준                                                                     |
| -------- | ------------------------------------------------------------------------ |
| High     | 금전, 정산, 권한, 데이터 정합성, 중복 결제, 운영 장애와 직접 관련된 영역 |
| Medium   | 사용자 불편, 운영 혼선, 상태 표시 오류와 관련된 영역                     |
| Low      | UI 문구, 정렬, 낮은 영향도의 사용성 개선 영역                            |

### High 우선순위 예시

- 결제 버튼 연속 클릭 시 중복 결제 발생 여부
- 결제 실패 주문이 정산에 반영되는지 여부
- 주문 취소 후 쿠폰/포인트 복원 여부
- 매장 A 관리자의 매장 B 주문 접근 가능 여부
- 일반 사용자의 관리자 API 직접 호출 가능 여부

### Medium 우선순위 예시

- 좌석 상태가 사용자 화면과 관리자 화면에서 다르게 표시되는 현상
- 품절 메뉴가 주문 가능하게 표시되는 현상
- 권한 없는 메뉴가 화면에 노출되는 현상

### Low 우선순위 예시

- 오류 메시지 문구가 다소 어색한 현상
- 버튼 간격, 색상, 정렬 등 시각적 사용성 문제
- 안내 문구 개선 사항

## 6. 테스트 레벨 정의

본 프로젝트에서는 테스트를 E2E 테스트, API 테스트, 수동 테스트, 회귀 테스트로 구분한다.

| 테스트 레벨 | 목적                                             | 적용 대상                                       | 예시                                        |
| ----------- | ------------------------------------------------ | ----------------------------------------------- | ------------------------------------------- |
| E2E 테스트  | 사용자 흐름 전체가 정상 동작하는지 검증          | 사용자 화면과 관리자 화면이 연결된 핵심 플로우  | 사용자 주문 → 관리자 접수 → 완료            |
| API 테스트  | 상태 변경, 권한, 정산 등 결과가 명확한 기능 검증 | 주문 상태 변경, 결제 실패, 정산 조회, 권한 차단 | 매장 A 관리자의 매장 B 주문 API 접근 시 403 |
| 수동 테스트 | 정성 판단이 필요한 영역 검증                     | UI/UX, 문구, 사용성, 시각적 요소                | 오류 메시지 자연스러움                      |
| 회귀 테스트 | 배포 전 핵심 기능 반복 검증                      | 로그인, 주문, 결제 실패, 권한 차단              | Playwright 자동 실행                        |

## 7. 테스트 유형별 접근 방식

### 7.1 E2E 테스트 접근 방식

E2E 테스트는 실제 사용자 흐름을 기준으로 검증한다.

본 프로젝트에서는 다음과 같은 흐름을 E2E 테스트 대상으로 우선 선정한다.

| E2E 후보                                  | 선정 이유                                       |
| ----------------------------------------- | ----------------------------------------------- |
| 로그인 후 좌석 선택                       | 서비스 이용 시작의 기본 흐름                    |
| 좌석에서 음식 주문                        | 사용자 핵심 행동                                |
| 사용자 주문 후 관리자 주문 목록 반영      | 사용자 화면과 관리자 화면 간 데이터 동기화 검증 |
| 관리자 주문 완료 처리 후 사용자 화면 반영 | 주문 상태 변경 흐름 검증                        |
| 일반 사용자의 관리자 페이지 접근 차단     | 화면 권한 검증                                  |

E2E 테스트는 실제 사용자의 흐름을 가장 잘 보여줄 수 있지만, 실행 속도가 느리고 유지보수 비용이 높다. 따라서 전체 테스트케이스를 E2E로 작성하지 않고, 핵심 사용자 플로우 위주로 제한한다.

---

### 7.2 API 테스트 접근 방식

API 테스트는 화면보다 빠르고 안정적으로 결과를 검증할 수 있는 영역에 적용한다.

특히 다음 영역은 API 테스트에 적합하다.

| API 테스트 대상    | 이유                                         |
| ------------------ | -------------------------------------------- |
| 주문 생성 API      | 주문 데이터 생성 여부를 명확히 확인 가능     |
| 주문 상태 변경 API | 상태 전이 규칙 검증 가능                     |
| 결제 실패 API      | 실패 시 주문 상태 유지 여부 검증 가능        |
| 쿠폰 복원 API      | 쿠폰 상태 변경 결과 검증 가능                |
| 정산 조회 API      | 완료 주문만 반영되는지 검증 가능             |
| 권한 검증 API      | 권한 없는 요청에 대한 401/403 응답 확인 가능 |

API 테스트는 권한, 상태 변경, 정산 반영처럼 기대 결과가 명확한 케이스에 우선 적용한다.

---

### 7.3 수동 테스트 접근 방식

수동 테스트는 사람의 판단이 필요한 영역에 적용한다.

| 수동 테스트 대상   | 이유                                                      |
| ------------------ | --------------------------------------------------------- |
| 오류 메시지 문구   | 사용자가 이해하기 쉬운지 판단 필요                        |
| 주문 화면 사용성   | 메뉴 선택 흐름이 자연스러운지 판단 필요                   |
| 관리자 화면 가독성 | 주문 상태와 좌석 상태를 빠르게 파악할 수 있는지 판단 필요 |
| 반응형 UI          | 다양한 화면 크기에서 사용성이 적절한지 판단 필요          |
| 시각적 정렬        | 자동화보다 사람의 시각적 판단이 효율적                    |

본 포트폴리오에서는 수동 테스트를 전면적으로 수행하지 않고, 자동화 대상에서 제외하는 기준으로만 정의한다.

---

### 7.4 회귀 테스트 접근 방식

회귀 테스트는 배포 전 반복 검증이 필요한 핵심 기능을 대상으로 한다.

회귀 테스트 후보는 다음과 같다.

| 회귀 테스트 후보       | 검증 목적               |
| ---------------------- | ----------------------- |
| 로그인 성공            | 모든 기능의 사전 조건   |
| 좌석 선택              | 서비스 이용 시작 검증   |
| 음식 주문 생성         | 핵심 비즈니스 흐름 검증 |
| 관리자 주문 처리       | 사용자-관리자 연동 검증 |
| 결제 실패 처리         | 금전 데이터 정합성 검증 |
| 주문 취소 후 쿠폰 복원 | 보상 데이터 정합성 검증 |
| 매장 간 권한 차단      | 운영 데이터 보호 검증   |
| 완료 주문 정산 반영    | 매출 데이터 신뢰성 검증 |

---

## 8. 자동화 전략

본 프로젝트는 모든 테스트케이스를 자동화하지 않는다.

자동화 대상은 다음 기준에 따라 선정한다.

1. 반복 실행 가치가 높은가?
2. 기대 결과가 명확한가?
3. 장애 발생 시 영향도가 큰가?
4. 수동 검증보다 자동화 효율이 높은가?
5. 테스트 데이터와 상태를 안정적으로 제어할 수 있는가?

### 자동화 분리 기준

| 구분        | 선정 기준                                    | 예시                             |
| ----------- | -------------------------------------------- | -------------------------------- |
| E2E 자동화  | 사용자 흐름 전체 검증이 필요한 경우          | 사용자 주문 → 관리자 접수 → 완료 |
| API 자동화  | 상태 변경, 권한, 정산처럼 결과가 명확한 경우 | 권한 없는 API 호출 시 403        |
| 수동 검증   | 사람의 판단이 필요한 경우                    | UI 문구, 사용성                  |
| 자동화 제외 | 변경 가능성이 높거나 자동화 가치가 낮은 경우 | 이벤트 배너, 일시적 프로모션     |

### 우선 자동화 대상

| 우선순위 | 자동화 대상                             | 테스트 유형 |
| -------- | --------------------------------------- | ----------- |
| 1        | 결제 실패 시 주문 상태 유지             | API         |
| 2        | 주문 취소 시 쿠폰/포인트 복원           | API         |
| 3        | 매장 A 관리자의 매장 B 데이터 접근 차단 | API         |
| 4        | 사용자 주문 후 관리자 화면 반영         | E2E         |
| 5        | 관리자 주문 완료 후 사용자 화면 반영    | E2E         |
| 6        | 완료 주문만 정산 반영                   | API         |
| 7        | 좌석 중복 선택 차단                     | E2E/API     |
| 8        | 일반 사용자의 관리자 페이지 접근 차단   | E2E/API     |

---

## 9. 테스트 데이터 전략

테스트 데이터는 권한, 매장, 좌석, 주문, 쿠폰, 결제 상태를 명확히 구분하여 사용한다.

| 데이터 유형 | 예시                                             | 목적                        |
| ----------- | ------------------------------------------------ | --------------------------- |
| 일반 사용자 | user01                                           | 좌석 선택, 주문, 결제 검증  |
| 매장 직원   | staff_store_a                                    | 주문 접수 및 상태 변경 검증 |
| 매장 관리자 | manager_store_a                                  | 담당 매장 정산 및 권한 검증 |
| 본사 관리자 | hq_admin                                         | 전체 매장 정산 조회 검증    |
| 매장        | Store A, Store B                                 | 매장 간 데이터 분리 검증    |
| 좌석        | A-01, A-02, B-01                                 | 좌석 상태 변경 검증         |
| 메뉴        | 김치볶음밥, 라면, 아이스티                       | 주문 생성 및 품절 검증      |
| 쿠폰        | 3,000원 할인 쿠폰                                | 쿠폰 사용/복원 검증         |
| 주문 상태   | 결제대기, 결제완료, 접수, 조리중, 완료, 취소완료 | 상태 전이 검증              |

### 테스트 데이터 관리 원칙

1. 테스트 계정은 권한별로 분리한다.
2. 매장 A와 매장 B 데이터를 분리하여 권한 검증에 사용한다.
3. 주문, 결제, 쿠폰 상태는 테스트 전후 초기화 가능해야 한다.
4. 중복 주문/중복 결제 테스트를 위해 동일 요청 반복 데이터를 준비한다.
5. 정산 검증을 위해 완료 주문, 취소 주문, 결제 실패 주문을 구분한다.
