# AI 테스트케이스 생성을 위한 요구사항 샘플

## 1. 문서 목적

본 문서는 AI 테스트케이스 생성을 위해 입력한 요구사항 샘플이다.

전체 요구사항을 한 번에 입력하기보다, 핵심 리스크가 큰 주문, 결제, 쿠폰, 정산, 권한 영역을 중심으로 AI에게 테스트케이스 초안을 생성하도록 요청했다.

---

## 2. 서비스 시나리오

PC방 매장 운영 플랫폼은 일반 사용자가 좌석을 선택하고 음식 주문과 결제를 진행하면, 매장 직원 또는 매장 관리자가 해당 주문을 접수하고 처리하는 서비스이다.

주문은 결제대기, 결제완료, 접수, 조리중, 완료, 취소완료, 환불완료 상태를 가진다.

정산은 완료 상태의 주문만 기준으로 집계한다.

---

## 3. 사용자 권한

| 권한        | 설명                                                 |
| ----------- | ---------------------------------------------------- |
| 일반 사용자 | 좌석 선택, 음식 주문, 결제, 본인 주문 내역 조회 가능 |
| 매장 직원   | 담당 매장의 주문 목록 조회 및 주문 상태 변경 가능    |
| 매장 관리자 | 담당 매장의 주문, 좌석, 매출, 정산 조회 가능         |
| 본사 관리자 | 전체 매장 정산 조회 및 쿠폰 정책 관리 가능           |

---

## 4. 핵심 요구사항

### 4.1 로그인/권한

- 사용자는 유효한 계정으로 로그인할 수 있다.
- 비로그인 사용자는 좌석 선택, 주문, 결제 화면에 접근할 수 없다.
- 일반 사용자는 관리자 페이지에 접근할 수 없다.
- 일반 사용자는 관리자 API를 호출할 수 없다.
- 매장 A 관리자는 매장 B 주문 또는 정산 데이터를 조회할 수 없다.
- 본사 관리자는 전체 매장 정산을 조회할 수 있다.

### 4.2 좌석

- 사용자는 사용 가능한 좌석을 선택할 수 있다.
- 좌석 선택 후 좌석 상태는 사용 가능에서 사용 중으로 변경된다.
- 사용 중인 좌석은 다른 사용자가 선택할 수 없다.
- 동일 좌석에 대한 동시 선택 요청이 발생하면 하나의 요청만 성공해야 한다.

### 4.3 주문

- 좌석을 사용 중인 사용자만 음식 주문을 생성할 수 있다.
- 품절 메뉴는 주문할 수 없다.
- 주문 버튼을 여러 번 클릭해도 주문은 1건만 생성되어야 한다.
- 주문 생성 후 관리자 주문 목록에 신규 주문이 표시되어야 한다.
- 사용자는 본인의 주문 내역만 조회할 수 있다.

### 4.4 주문 상태

주문 상태는 아래 순서에 따라 변경된다.

```text
payment_pending
→ payment_completed
→ accepted
→ cooking
→ completed
```
