# PC방 매장 운영 플랫폼 서비스 개요

## 1. 프로젝트 개요

본 프로젝트는 PC방 매장 운영 플랫폼을 가정하여 회원, 좌석, 음식 주문, 결제, 쿠폰, 정산, 관리자 권한 흐름을 검증하기 위한 QA 포트폴리오 프로젝트이다.

본 포트폴리오의 목적은 단순한 화면 테스트 자동화가 아니라, 실제 매장 운영에 영향을 줄 수 있는 품질 리스크를 식별하고, 요구사항 분석부터 테스트케이스 설계, 자동화 대상 선정, Playwright 기반 E2E/API 테스트까지의 QA 프로세스를 보여주는 것이다.

## 2. 서비스 목적

PC방 매장 운영 플랫폼은 사용자가 좌석을 선택하고 음식 주문 및 결제를 진행할 수 있으며, 매장 관리자는 주문 처리와 좌석 상태를 관리하고, 본사 관리자는 매장별 매출과 정산 현황을 확인할 수 있는 운영 관리 시스템이다.

## 3. 주요 사용자

서비스 사용자는 크게 네 가지 권한으로 구분된다.

| 사용자 유형 | 설명                                                    |
| ----------- | ------------------------------------------------------- |
| 일반 사용자 | PC방을 이용하는 고객                                    |
| 매장 직원   | 주문 접수 및 좌석 상태를 관리하는 직원                  |
| 매장 관리자 | 매장 운영, 매출, 직원, 정산을 관리하는 관리자           |
| 본사 관리자 | 전체 매장 현황과 통합 정산, 쿠폰 정책을 관리하는 관리자 |

## 4. 주요 기능 범위

| 영역             | 주요 기능                                         |
| ---------------- | ------------------------------------------------- |
| 회원/로그인      | 로그인, 로그아웃, 권한별 메뉴 노출                |
| 좌석             | 좌석 선택, 좌석 사용 시작, 좌석 상태 변경         |
| 주문             | 음식 메뉴 조회, 주문 생성, 주문 상태 확인         |
| 결제             | 결제 성공, 결제 실패, 중복 결제 방지              |
| 쿠폰/포인트      | 쿠폰 사용, 포인트 적립, 주문 취소 시 복원         |
| 관리자 주문 처리 | 주문 접수, 조리 중, 완료 상태 변경                |
| 정산             | 완료 주문 기준 매출 반영, 일 매출 조회            |
| 권한             | 사용자 유형별 접근 제어, 매장 간 데이터 접근 제한 |

## 5. QA 관점에서 중요한 특징

본 서비스는 단순 조회형 서비스가 아니라, 여러 상태 값이 계속 변경되는 운영 플랫폼이다.

따라서 QA 검증 시 다음 관점이 중요하다.

1. 좌석, 주문, 결제, 쿠폰, 정산 상태가 일관되게 변경되는지 확인해야 한다.
2. 사용자 화면과 관리자 화면 간 데이터가 동일하게 반영되는지 확인해야 한다.
3. 결제 실패, 주문 취소, 품절, 중복 클릭과 같은 예외 상황을 검증해야 한다.
4. 매장 A 관리자가 매장 B 데이터에 접근할 수 없는지 확인해야 한다.
5. 완료된 주문만 정산에 반영되는지 확인해야 한다.

## 6. 포트폴리오에서 다룰 핵심 플로우

본 포트폴리오는 전체 기능을 모두 다루기보다, 운영 리스크가 큰 핵심 플로우를 중심으로 검증한다.

| 번호 | 핵심 플로우         | 검증 목적                                           |
| ---- | ------------------- | --------------------------------------------------- |
| 1    | 로그인 및 좌석 선택 | 사용자가 좌석을 정상적으로 사용할 수 있는지 확인    |
| 2    | 음식 주문           | 사용자 주문이 관리자 화면에 정상 반영되는지 확인    |
| 3    | 결제 실패/취소      | 결제 실패와 취소 시 데이터 정합성이 유지되는지 확인 |
| 4    | 권한별 접근 제어    | 사용자 유형별 접근 제한이 정상 동작하는지 확인      |
| 5    | 정산 반영           | 완료된 주문만 매출에 반영되는지 확인                |

## 7. 테스트 대상의 가정

본 포트폴리오에서는 실제 상용 서비스를 구현하지 않고, QA 검증 목적의 mock 서비스 또는 테스트용 화면을 대상으로 한다.

테스트 대상은 다음 중 하나로 구성할 수 있다.

- 간단한 HTML mock 화면
- React 기반 mock 서비스
- JSON mock API
- Playwright 테스트용 샘플 애플리케이션

중요한 것은 서비스 자체의 개발 완성도가 아니라, QA가 요구사항을 분석하고 리스크 기반으로 테스트를 설계하며 자동화 대상을 선정하는 과정이다.
