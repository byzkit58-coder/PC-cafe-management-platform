# 자동화 대상 선정

## 1. 문서 목적

본 문서는 PC방 매장 운영 플랫폼 QA 포트폴리오의 테스트케이스 중 Playwright 자동화 대상으로 선정할 항목을 정의한다.

자동화 대상은 전체 테스트케이스를 무조건 자동화하는 방식이 아니라, 운영 영향도, 반복 검증 가치, 기대 결과 명확성, 테스트 데이터 제어 가능성, 자동화 유지보수성을 기준으로 선정한다.

본 문서는 이후 Playwright E2E/API 테스트 구현 범위와 GitHub Actions 기반 회귀 테스트 구성의 기준으로 사용된다.

---

## 2. 자동화 선정 기준

자동화 대상은 다음 기준에 따라 선정한다.

| 기준 | 설명 |
|---|---|
| 운영 영향도 | 실패 시 사용자, 매장 운영, 결제, 정산, 권한에 큰 영향이 있는가 |
| 반복 검증 가치 | 배포 전마다 반복적으로 확인해야 하는가 |
| 기대 결과 명확성 | 자동화 assertion으로 Pass/Fail 판단이 명확한가 |
| 테스트 데이터 제어 가능성 | 테스트 전후 데이터를 안정적으로 준비하고 초기화할 수 있는가 |
| 자동화 안정성 | 화면 구조 변경에 지나치게 취약하지 않은가 |
| 수동 대비 효율 | 사람이 반복 확인하는 것보다 자동화 효율이 높은가 |

---

## 3. 자동화 유형 분류 기준

### 3.1 E2E 자동화

E2E 자동화는 사용자 화면과 관리자 화면이 연결되는 실제 업무 흐름을 검증하는 데 사용한다.

| 선정 기준 | 예시 |
|---|---|
| 사용자 흐름 전체 검증 필요 | 로그인 → 좌석 선택 → 주문 |
| 화면 간 데이터 동기화 확인 필요 | 사용자 주문 → 관리자 주문 목록 반영 |
| 실제 클릭/화면 이동 기반 검증 필요 | 주문 버튼 연속 클릭 |
| 화면 권한 접근 확인 필요 | 일반 사용자 관리자 페이지 접근 차단 |

---

### 3.2 API 자동화

API 자동화는 상태 변경, 권한, 정산, 데이터 정합성처럼 결과가 명확한 영역에 사용한다.

| 선정 기준 | 예시 |
|---|---|
| 응답 코드 검증이 핵심 | 권한 없는 API 호출 시 403 |
| 상태 값 검증이 핵심 | 결제 실패 시 주문 상태 결제대기 유지 |
| 데이터 정합성 검증 필요 | 주문 취소 후 쿠폰/포인트 복원 |
| 정산 결과 검증 필요 | 완료 주문만 일 매출 반영 |

---

### 3.3 수동 검증

수동 검증은 자동화보다 사람의 판단이 더 적합한 영역에 사용한다.

| 선정 기준 | 예시 |
|---|---|
| 문구 자연스러움 판단 필요 | 오류 메시지 |
| 시각적 가독성 판단 필요 | 관리자 주문 목록 |
| 사용성 판단 필요 | 주문 화면 흐름 |
| 정책 변경 가능성 높음 | 이벤트성 쿠폰 안내 문구 |

---

### 3.4 자동화 제외

자동화 제외 대상은 1차 포트폴리오 범위에서 구현 비용이 높거나 자동화 가치가 낮은 영역으로 정의한다.

| 제외 기준 | 예시 |
|---|---|
| 외부 시스템 의존 | 실제 PG 결제 승인 |
| 실제 장비 필요 | PC방 좌석 하드웨어 연동 |
| 1차 포트폴리오 범위 초과 | 부하 테스트, 전체 보안 진단 |
| 변경 가능성 높음 | 이벤트 배너, 임시 프로모션 |

---

## 4. 자동화 점수 평가 기준

자동화 우선순위는 각 항목을 1~5점으로 평가하여 산정한다.

| 평가 항목 | 설명 |
|---|---|
| 운영 영향도 | 장애 발생 시 비즈니스 영향이 클수록 높은 점수 |
| 반복 검증 가치 | 배포 전 반복 확인 필요성이 높을수록 높은 점수 |
| 기대 결과 명확성 | assertion으로 결과 확인이 쉬울수록 높은 점수 |
| 데이터 제어 가능성 | 테스트 데이터 setup/teardown이 쉬울수록 높은 점수 |
| 자동화 안정성 | UI 변경이나 환경 변화에 덜 취약할수록 높은 점수 |

| 총점 | 판단 |
|---:|---|
| 20점 이상 | 우선 자동화 |
| 16~19점 | 자동화 후보 |
| 11~15점 | 수동 또는 후순위 |
| 10점 이하 | 자동화 제외 |

## 8. 자동화 제외 대상

| 제외 대상 | 제외 사유 |
|---|---|
| 실제 PG 결제 승인/취소 | 외부 결제 시스템 연동이 필요하므로 mock 결제로 대체 |
| 실제 PC방 좌석 하드웨어 연동 | 실제 장비가 필요하므로 포트폴리오 범위에서 제외 |
| 전체 성능 테스트 | 1차 포트폴리오 목적은 자동화 전략과 회귀 테스트 구성 |
| 전체 보안 취약점 진단 | 권한 접근 제어 검증 수준으로 제한 |
| 이벤트 배너/프로모션 문구 | 변경 가능성이 높고 자동화 가치가 낮음 |

## 10. 결론

본 프로젝트의 자동화 대상은 단순히 구현 가능한 테스트를 선택한 것이 아니라, 운영 리스크와 반복 검증 가치를 기준으로 선정했다.

사용자 화면과 관리자 화면이 연결되는 핵심 흐름은 E2E 테스트로 자동화하고, 결제 실패, 정산 반영, 쿠폰/포인트 복원, 권한 차단처럼 결과가 명확한 영역은 API 테스트로 자동화한다.

수동 검증이 더 적합한 오류 메시지, 화면 가독성, 사용성 영역은 자동화 대상에서 제외하거나 후순위로 분류한다.

이를 통해 본 포트폴리오는 테스트 자동화 구현뿐 아니라, 자동화 우선순위를 판단하고 범위를 설계하는 QA 역량을 보여주는 것을 목표로 한다.
