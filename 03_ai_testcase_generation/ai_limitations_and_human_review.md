# AI 생성 테스트케이스 한계 및 QA 리뷰 결과

## 1. 문서 목적

본 문서는 AI가 생성한 테스트케이스 초안을 QA 관점에서 검토한 결과와 보완 내용을 정리한다.

본 포트폴리오에서는 AI를 테스트 설계 보조 도구로 활용하되, 생성 결과를 그대로 사용하지 않고 QA가 요구사항, 리스크, 데이터 정합성, 권한, 자동화 적합도를 기준으로 검토했다.

---

## 2. AI 활용 목적

AI는 다음 목적으로 활용했다.

1. 요구사항 기반 테스트케이스 초안 생성
2. 정상/예외 케이스의 빠른 도출
3. 테스트케이스 작성 형식 표준화
4. 자동화 적합도 초안 분류
5. QA 리뷰 전 검토 대상 확보

AI 활용의 목적은 QA 판단을 대체하는 것이 아니라, 테스트 설계 초기 초안을 빠르게 확보하고 QA가 누락 리스크를 보완하는 것이다.

---

## 3. AI가 잘 생성한 영역

AI는 다음 영역에서 유용한 초안을 생성했다.

| 영역                    | 평가                                             |
| ----------------------- | ------------------------------------------------ |
| 로그인 정상/실패 케이스 | 기본적인 인증 흐름을 적절히 생성                 |
| 좌석 선택 정상 케이스   | 사용 가능한 좌석 선택과 사용 중 좌석 차단을 포함 |
| 음식 주문 정상 케이스   | 좌석 사용 중 주문 생성 흐름을 포함               |
| 결제 성공/실패 케이스   | 결제 결과에 따른 주문 상태 변경을 포함           |
| 쿠폰 복원 케이스        | 주문 취소 후 쿠폰 복원 검증을 포함               |
| 권한 접근 차단 케이스   | 일반 사용자와 매장 관리자 권한 차이를 일부 반영  |
| 수동 테스트 분류        | 오류 메시지 등 Manual 대상 구분 가능             |

---

## 4. AI 결과의 주요 한계

AI 생성 결과에는 다음과 같은 한계가 있었다.

| 한계 ID   | 한계 내용                                                                            | 영향                                 |
| --------- | ------------------------------------------------------------------------------------ | ------------------------------------ |
| LIMIT-001 | 화면 접근 권한 검증은 포함했으나 API 직접 호출 검증이 부족했다.                      | 실제 보안 결함을 놓칠 수 있음        |
| LIMIT-002 | 매장 A/B 간 데이터 분리를 주문 영역 위주로만 다뤘다.                                 | 정산 데이터 유출 리스크 누락         |
| LIMIT-003 | 결제 실패 시 쿠폰/포인트 상태 유지 검증이 부족했다.                                  | 사용자 혜택 데이터 손실 가능         |
| LIMIT-004 | 완료 주문만 정산 반영되는 조건은 포함했으나 결제실패/환불 상태 정산 제외가 부족했다. | 매출 데이터 오류 가능                |
| LIMIT-005 | 주문 상태 전이의 비정상 경로 차단 검증이 부족했다.                                   | 잘못된 상태 변경 가능                |
| LIMIT-006 | 중복 결제는 포함했으나 중복 주문 생성 방지 케이스가 약했다.                          | 운영 혼선 및 CS 발생 가능            |
| LIMIT-007 | 동시성 상황에 대한 검증이 부족했다.                                                  | 좌석 중복 선택 문제를 놓칠 수 있음   |
| LIMIT-008 | 자동화 적합도 분류가 일부 넓게 지정되었다.                                           | E2E/API 역할 구분이 모호해질 수 있음 |

---

## 5. QA 보완 방향

AI 결과를 바탕으로 QA는 다음 방향으로 테스트케이스를 보완했다.

| 보완 방향                   | 설명                                                       |
| --------------------------- | ---------------------------------------------------------- |
| API 권한 검증 강화          | 화면 차단뿐 아니라 API 직접 호출 시 401/403 응답 검증 추가 |
| 매장 간 데이터 분리 강화    | 주문 데이터와 정산 데이터 모두 매장별로 분리 검증          |
| 결제 실패 후 상태 검증 확대 | 주문 상태, 쿠폰, 포인트, 정산 반영 여부를 함께 확인        |
| 정산 조건 세분화            | 완료, 결제실패, 취소완료, 환불완료 상태별 반영 여부 분리   |
| 상태 전이 제한 추가         | 결제대기 → 완료, 취소완료 → 조리중 등 비정상 변경 차단     |
| 중복 요청 검증 추가         | 중복 주문, 중복 결제, 동일 좌석 동시 선택 검증             |
| 자동화 유형 재분류          | 화면 흐름은 E2E, 상태/권한/정산은 API 중심으로 재분류      |

## 6. AI 결과와 QA 보완 비교

| 비교 항목 | AI 생성 결과               | QA 보완 결과                                      |
| --------- | -------------------------- | ------------------------------------------------- |
| 로그인    | 정상/실패 로그인 포함      | 비로그인 보호 페이지 접근 차단 강화               |
| 좌석      | 좌석 선택/중복 선택 포함   | 동일 좌석 동시 선택 요청 검증 추가                |
| 주문      | 주문 생성/관리자 반영 포함 | 중복 주문 방지, 본인 주문 조회 제한 추가          |
| 결제      | 성공/실패/중복 결제 포함   | 실패 시 쿠폰/포인트/정산 영향까지 확대            |
| 쿠폰      | 주문 취소 후 복원 포함     | 결제 실패 시 사용 처리 방지 추가                  |
| 포인트    | 주문 취소 후 복원 포함     | 결제 실패 시 차감 방지 추가                       |
| 정산      | 완료/취소 주문 중심        | 결제실패/환불완료/매장별 분리/할인 금액 기준 추가 |
| 권한      | 관리자 접근 차단 포함      | API 직접 호출, 매장 간 정산 접근 차단 추가        |
| 상태 전이 | 정상 상태 변경 중심        | 비정상 상태 변경 차단 추가                        |
| 자동화    | E2E/API 혼합               | 검증 목적에 따라 E2E/API/Manual 재분류            |

## 7. QA 최종 판단

AI 테스트케이스 초안은 전체 테스트 설계의 출발점으로는 유용했다.

다만 AI 결과는 일반적인 정상 흐름과 대표 예외 케이스에 집중되는 경향이 있었고, 실제 운영 플랫폼에서 중요한 다음 영역은 부족했다.

1. API 직접 호출 권한 검증
2. 매장 간 정산 데이터 접근 차단
3. 결제 실패 후 쿠폰/포인트 상태 유지
4. 정산 반영 조건 세분화
5. 비정상 주문 상태 전이 차단
6. 중복 주문/중복 결제 방지
7. 동시성 기반 좌석 중복 선택 차단

따라서 최종 테스트케이스는 AI 결과를 그대로 사용하지 않고, QA 리뷰를 통해 리스크 기반으로 보완했다.

## 8. 자동화 대상 선정에 반영한 내용

QA 리뷰 후 자동화 대상 선정 기준은 다음과 같이 조정했다.

| 영역                          | 자동화 방향        | 이유                                                  |
| ----------------------------- | ------------------ | ----------------------------------------------------- |
| 결제 실패                     | API 자동화         | 주문 상태, 쿠폰, 포인트, 정산 결과를 빠르게 검증 가능 |
| 쿠폰/포인트 복원              | API 자동화         | 상태 값 검증이 명확함                                 |
| 정산 반영                     | API 자동화         | 완료/취소/실패/환불 상태별 결과 검증 가능             |
| 권한 차단                     | API 우선, 일부 E2E | API 직접 호출 차단이 핵심                             |
| 주문 생성 후 관리자 반영      | E2E 자동화         | 사용자 화면과 관리자 화면 연동 검증 필요              |
| 주문 상태 변경 후 사용자 반영 | E2E 자동화         | 화면 간 동기화 확인 필요                              |
| 오류 메시지                   | 수동 검증          | 문구와 사용성 판단 필요                               |
| 시각적 가독성                 | 수동 검증          | 자동화보다 사람 판단이 적합                           |
