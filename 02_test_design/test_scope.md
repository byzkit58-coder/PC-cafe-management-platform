# 테스트 범위

## 1. 문서 목적

본 문서는 PC방 매장 운영 플랫폼 QA 포트폴리오에서 테스트에 포함할 범위와 제외할 범위를 정의한다.

포트폴리오의 목적은 모든 기능을 완벽하게 구현하고 검증하는 것이 아니라, 경력직 QA 관점에서 핵심 운영 리스크를 식별하고, 테스트 전략과 자동화 대상을 합리적으로 선정하는 과정을 보여주는 것이다.

따라서 본 문서는 테스트 범위를 명확히 제한하여 포트폴리오가 과도하게 확장되는 것을 방지한다.

| 영역 | 포함 여부 | 설명 |

| 음식 주문 | 포함 | 사용자 핵심 행동 및 관리자 주문 처리와 연결 |
| 주문 상태 변경 | 포함 | 결제대기, 결제완료, 접수, 조리중, 완료, 취소완료 검증 |
| 결제 성공/실패 | 포함 | 금전 데이터 및 주문 상태 정합성 검증 |
| 중복 결제 방지 | 포함 | 결제 버튼 연속 클릭 및 동일 주문 중복 결제 방지 |
| 중복 주문 방지 | 포함 | 주문 버튼 연속 클릭 시 주문 1건만 생성되는지 검증 |
| 쿠폰 사용/복원 | 포함 | 주문 취소 및 결제 실패 시 쿠폰 상태 검증 |
| 포인트 사용/복원 | 포함 | 주문 취소 시 포인트 복원 검증 |
| 관리자 주문 처리 | 포함 | 주문 접수, 조리중, 완료 처리 검증 |
| 정산 반영 | 포함 | 완료 주문만 일 매출에 반영되는지 검증 |
| 권한별 접근 제어 | 포함 | 일반 사용자, 매장 직원, 매장 관리자, 본사 관리자 권한 검증 |
| 매장 간 데이터 접근 제한 | 포함 | 매장 A 관리자의 매장 B 데이터 접근 차단 검증 |
| CI 리포트 | 포함 | Playwright 테스트 실행 결과 증빙 |
| 결함 리포트 | 포함 | 발견 가능한 주요 결함 사례 문서화 |

## 3. 테스트 제외 범위

본 포트폴리오에서는 다음 영역을 테스트 범위에서 제외한다.

| 제외 영역 | 제외 사유 |
|---|---|
| 실제 PG 결제 연동 | 포트폴리오 범위상 mock 결제 응답으로 대체 |
| 실제 카드 승인/취소/환불 처리 | 외부 결제 시스템 연동 필요 |
| 실제 회원가입 및 본인인증 | 핵심 QA 시나리오와 직접 관련 낮음 |
| 실시간 좌석 하드웨어 연동 | 실제 PC방 장비 및 좌석 제어 시스템 필요 |
| 전체 재고 관리 기능 | 품절 메뉴 주문 차단 수준으로 제한 |
| 본사 관리자 전체 기능 | 정산 조회와 쿠폰 정책 관리 개념 수준으로 제한 |
| 쿠폰 정책 생성 전체 플로우 | 쿠폰 사용/복원 검증 중심으로 제한 |
| 직원 관리 상세 기능 | 권한 검증 범위에 포함하지 않음 |
| 모바일 앱 테스트 | Web QA 포트폴리오 중심으로 제한 |
| 크로스 브라우저 전체 검증 | Chromium 기준으로 제한 |
| 성능 테스트 | 1차 포트폴리오 범위에서는 제외 |
| 부하 테스트 | 실제 운영 트래픽 환경이 아니므로 제외 |
| 보안 취약점 진단 전체 | 권한 접근 제어 수준으로 제한 |
| 접근성 테스트 전체 | 본 포트폴리오 핵심 범위에서 제외 |
| 운영 모니터링/알림 | 테스트 자동화 포트폴리오 범위 밖으로 제외 |

## 5. 테스트 유형별 범위

## 5.1 E2E 테스트 범위

E2E 테스트는 사용자 화면과 관리자 화면이 연결되는 핵심 흐름을 대상으로 한다.

| E2E 테스트 대상 | 포함 이유 |
|---|---|
| 일반 사용자 로그인 | 모든 사용자 기능의 시작점 |
| 좌석 선택 | PC방 이용 시작 플로우 |
| 좌석에서 음식 주문 | 핵심 사용자 행동 |
| 사용자 주문 후 관리자 주문 목록 반영 | 사용자-관리자 데이터 연동 검증 |
| 관리자 주문 상태 변경 | 주문 처리 흐름 검증 |
| 관리자 상태 변경 후 사용자 화면 반영 | 화면 간 동기화 검증 |
| 일반 사용자 관리자 페이지 접근 차단 | 화면 권한 검증 |
| 주문 버튼 연속 클릭 시 주문 1건만 생성 | 실제 사용자 행동 기반 예외 검증 |

---

## 5.2 API 테스트 범위

API 테스트는 상태 변경, 권한, 정산, 데이터 정합성처럼 결과가 명확한 영역을 대상으로 한다.

| API 테스트 대상 | 포함 이유 |
|---|---|
| 주문 생성 API | 주문 데이터 생성 검증 |
| 주문 상태 변경 API | 상태 전이 규칙 검증 |
| 결제 성공/실패 API | 결제 결과에 따른 주문 상태 검증 |
| 결제 중복 요청 API | 중복 결제 방지 검증 |
| 주문 취소 API | 주문 취소 상태 변경 검증 |
| 쿠폰 복원 API | 쿠폰 상태 정합성 검증 |
| 포인트 복원 API | 포인트 상태 정합성 검증 |
| 정산 조회 API | 완료 주문만 반영되는지 검증 |
| 매장별 주문 조회 API | 매장 간 권한 분리 검증 |
| 관리자 API 권한 검증 | 권한 없는 사용자 접근 차단 |

## 5.3 수동 테스트 범위

수동 테스트는 자동화보다 사람의 판단이 더 적합한 영역에 한정한다.

| 수동 테스트 대상 | 포함 이유 |
|---|---|
| 오류 메시지 문구 | 사용자가 이해하기 쉬운지 판단 필요 |
| 주문 화면 사용성 | 메뉴 탐색과 주문 흐름이 자연스러운지 판단 필요 |
| 관리자 주문 목록 가독성 | 운영자가 빠르게 주문 상태를 파악할 수 있는지 확인 |
| 좌석 현황 화면 가독성 | 좌석 상태 구분이 명확한지 확인 |
| 반응형 화면 확인 | 화면 크기에 따른 사용성 판단 필요 |
| 시각적 정렬 | 자동화보다 수동 확인이 효율적 |

## 7. 브라우저/디바이스 범위

본 포트폴리오는 1차 범위에서 Chromium 기반 Web 테스트를 기준으로 한다.

| 항목 | 포함 여부 | 설명 |
|---|---|---|
| Chromium | 포함 | 기본 자동화 대상 |
| Firefox | 제외 | 1차 포트폴리오 범위에서는 제외 |
| WebKit | 제외 | 1차 포트폴리오 범위에서는 제외 |
| Mobile Web | 제외 | Web QA 포트폴리오 기본 범위 이후 확장 가능 |
| Native App | 제외 | 현재 포트폴리오는 Web 테스트 중심 |

브라우저 범위를 Chromium으로 제한하는 이유는 포트폴리오의 핵심 목적이 크로스 브라우저 호환성 검증이 아니라, QA 전략과 자동화 설계 역량을 보여주는 것이기 때문이다.

## 9. 테스트 데이터 초기화 기준

테스트 자동화의 안정성을 위해 다음 기준을 적용한다.

1. 각 테스트는 가능한 독립적으로 실행될 수 있어야 한다.
2. 주문, 결제, 쿠폰, 포인트 데이터는 테스트 전 초기 상태를 보장해야 한다.
3. 테스트 실행 후 생성된 주문 데이터는 삭제하거나 상태를 초기화한다.
4. 권한 검증용 계정은 테스트 중 권한이 변경되지 않는 고정 계정을 사용한다.
5. 매장 A와 매장 B 데이터는 명확히 분리한다.
6. 정산 검증을 위해 완료 주문, 취소 주문, 결제 실패 주문 데이터를 구분한다.

## 10. 범위 제한 사유

본 포트폴리오는 경력직 QA 역량을 보여주는 것이 목적이므로, 모든 운영 기능을 구현하고 검증하는 방식은 적합하지 않다.

따라서 다음 기준에 따라 범위를 제한한다.

1. 실제 외부 시스템 연동이 필요한 기능은 mock 처리한다.
2. QA 핵심 역량과 직접 관련이 낮은 기능은 제외한다.
3. 자동화 가치가 낮거나 변경 가능성이 높은 기능은 제외한다.
4. 1차 포트폴리오에서는 Web 기반 핵심 회귀 테스트에 집중한다.
5. 성능, 접근성, 보안 진단 전체는 별도 확장 영역으로 남긴다.

## 12. 테스트 범위 결론

본 프로젝트의 테스트 범위는 PC방 매장 운영 플랫폼의 핵심 운영 리스크를 검증하는 데 집중한다.

포함 범위는 로그인, 좌석, 주문, 결제, 쿠폰/포인트, 정산, 권한 검증으로 제한한다.

제외 범위는 실제 PG 연동, 하드웨어 연동, 성능 테스트, 전체 보안 진단, 모바일 앱 테스트 등 포트폴리오 핵심 목적과 직접 관련이 낮거나 구현 비용이 큰 영역으로 정의한다.

이를 통해 본 포트폴리오는 단순 기능 나열이 아니라, 리스크 기반 테스트 전략과 자동화 우선순위 선정 능력을 보여주는 것을 목표로 한다.
