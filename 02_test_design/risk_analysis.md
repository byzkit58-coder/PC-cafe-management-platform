# 리스크 분석

## 1. 문서 목적

본 문서는 PC방 매장 운영 플랫폼의 주요 품질 리스크를 식별하고, 리스크별 우선순위와 대응 테스트 전략을 정의한다.

본 프로젝트는 단순 기능 검증이 아니라 실제 운영에 영향을 줄 수 있는 리스크를 중심으로 테스트를 설계한다.

특히 금전 데이터, 정산, 쿠폰/포인트, 권한, 주문 상태 변경과 관련된 리스크를 High 우선순위로 분류한다.

---

## 2. 리스크 평가 기준

리스크는 다음 세 가지 기준으로 평가한다.

| 평가 항목 | 설명 |
|---|---|
| 영향도 | 문제가 발생했을 때 사용자, 매장 운영, 정산, 보안에 미치는 영향 |
| 발생 가능성 | 실제 사용 중 해당 문제가 발생할 가능성 |
| 탐지 난이도 | 문제가 발생했을 때 QA 또는 운영자가 발견하기 쉬운 정도 |

---

## 3. 리스크 등급 기준

| 등급 | 기준 |
|---|---|
| High | 금전 손실, 정산 오류, 권한 오류, 데이터 유출, 중복 결제, 핵심 운영 장애와 관련된 리스크 |
| Medium | 사용자 불편, 운영 혼선, 상태 표시 오류, 일부 기능 사용 제한과 관련된 리스크 |
| Low | UI 문구, 정렬, 낮은 영향도의 사용성 문제와 관련된 리스크 |

## 5. 기능별 리스크 분석

## 5.1 회원/로그인 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 비로그인 사용자가 보호 페이지에 접근 | 주문/좌석 기능 오남용 가능 |
| 로그아웃 후 기존 세션으로 접근 가능 | 인증 세션 관리 오류 |
| 권한별 메뉴가 잘못 노출 | 권한 혼선 및 보안 리스크 |

### 분석

로그인 기능은 모든 사용자 흐름의 시작점이다.  
단순 로그인 성공 여부보다 중요한 것은 인증 상태와 권한 상태가 이후 기능 접근에 정확히 반영되는지이다.

특히 일반 사용자가 관리자 메뉴를 보거나, 로그아웃 후에도 보호 페이지에 접근할 수 있다면 권한 제어 결함으로 이어질 수 있다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 유효한 계정 로그인 성공 | E2E | High |
| 비로그인 사용자의 좌석/주문 접근 차단 | E2E | High |
| 로그아웃 후 보호 페이지 접근 차단 | E2E/API | Medium |
| 권한별 메뉴 노출 확인 | E2E/Manual | Medium |

## 5.2 좌석 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 사용 중 좌석 중복 선택 | 매장 운영 혼선 |
| 좌석 상태가 사용자/관리자 화면에서 불일치 | 운영자 판단 오류 |
| 동시 요청 시 좌석 상태 중복 변경 | 데이터 정합성 저하 |

### 분석

좌석 상태는 PC방 운영의 기본 데이터이다.  
좌석이 사용 중임에도 다른 사용자가 선택할 수 있거나, 사용자 화면과 관리자 화면의 좌석 상태가 다르게 표시되면 실제 매장 운영에 혼선이 발생할 수 있다.

다만 좌석 오류는 금전 데이터나 보안 문제보다는 영향도가 낮으므로 대부분 Medium 리스크로 분류한다.  
단, 동일 좌석 동시 선택과 같이 데이터 정합성 문제가 큰 경우에는 API 테스트로 별도 검증한다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 사용 가능한 좌석 선택 성공 | E2E | High |
| 사용 중 좌석 선택 차단 | E2E/API | Medium |
| 동일 좌석 동시 선택 요청 시 1건만 성공 | API | Medium |
| 관리자 화면 좌석 상태 반영 확인 | E2E | Medium |

## 5.3 주문 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 주문 생성 실패 | 핵심 사용자 흐름 장애 |
| 사용자 주문이 관리자 화면에 미표시 | 매장 주문 누락 |
| 주문 버튼 중복 클릭으로 중복 주문 생성 | 운영 혼선 및 CS 발생 |
| 품절 메뉴 주문 가능 | 재고 관리 오류 및 사용자 불만 |

### 분석

주문은 사용자 화면과 관리자 화면이 연결되는 핵심 흐름이다.  
사용자 주문이 관리자 화면에 표시되지 않으면 실제 매장에서 주문을 처리할 수 없으므로 High 리스크로 분류한다.

또한 주문 버튼 연속 클릭으로 주문이 여러 건 생성되면 중복 조리, 중복 결제, 취소 요청 등 운영 부담이 발생할 수 있다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 좌석 사용 중 음식 주문 성공 | E2E | High |
| 주문 후 관리자 주문 목록 반영 | E2E | High |
| 주문 버튼 연속 클릭 시 주문 1건만 생성 | E2E/API | High |
| 품절 메뉴 주문 차단 | E2E/API | Medium |
| 좌석 미사용 사용자의 주문 차단 | API | Medium |

## 5.4 결제 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 중복 결제 발생 | 금전 손실 및 CS 발생 |
| 결제 실패 주문이 결제완료 처리 | 데이터 정합성 오류 |
| 결제 실패 주문이 정산에 반영 | 매출 데이터 오류 |
| 결제 성공 후 주문 상태 미변경 | 주문 처리 지연 |

### 분석

결제 영역은 금전 데이터와 직접 연결되므로 최우선 검증 영역이다.

특히 결제 버튼을 연속 클릭했을 때 중복 결제가 발생하거나, 결제 실패 주문이 정산에 포함되면 사용자 피해와 매장 매출 데이터 오류로 이어질 수 있다.

따라서 결제 성공보다 결제 실패, 중복 요청, 정산 반영 여부를 더 중요하게 검증한다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 결제 성공 시 주문 상태 결제완료 변경 | API/E2E | High |
| 결제 실패 시 주문 상태 결제대기 유지 | API | High |
| 결제 실패 주문 정산 미반영 | API | High |
| 결제 버튼 연속 클릭 시 결제 1건만 생성 | API/E2E | High |
| 동일 주문 중복 결제 차단 | API | High |

## 5.5 쿠폰/포인트 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 주문 취소 후 쿠폰 미복원 | 사용자 보상 문제 |
| 결제 실패 시 쿠폰이 사용 처리됨 | 데이터 정합성 오류 |
| 포인트 사용 후 취소 시 미복원 | 사용자 금전성 혜택 손실 |
| 사용 완료 쿠폰 재사용 가능 | 정책 위반 |

### 분석

쿠폰과 포인트는 사용자의 금전성 혜택과 연결된다.  
금액 자체는 결제보다 작을 수 있지만, 사용자 불만과 CS 발생 가능성이 높다.

특히 주문 취소, 결제 실패, 환불 상황에서 쿠폰/포인트 상태가 정확히 복원되는지 검증해야 한다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 쿠폰 적용 주문 생성 | API/E2E | High |
| 주문 취소 시 쿠폰 복원 | API | High |
| 결제 실패 시 쿠폰 미사용 유지 | API | High |
| 주문 취소 시 포인트 복원 | API | High |
| 사용 완료 쿠폰 재사용 차단 | API | Medium |

## 5.6 관리자 주문 처리 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 관리자 주문 목록에 주문 미표시 | 주문 처리 누락 |
| 주문 상태 변경 순서 오류 | 운영 프로세스 혼선 |
| 취소 주문을 조리중/완료로 변경 가능 | 데이터 정합성 오류 |
| 관리자 변경 상태가 사용자 화면에 미반영 | 사용자 혼란 |

### 분석

관리자 주문 처리는 사용자 주문이 실제 매장 운영으로 이어지는 구간이다.

주문 상태가 정책에 맞지 않는 순서로 변경되거나, 관리자 화면에서 변경한 상태가 사용자 화면에 반영되지 않으면 운영 신뢰도가 낮아진다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 신규 주문 관리자 목록 표시 | E2E | High |
| 결제완료 → 접수 → 조리중 → 완료 상태 변경 | E2E/API | High |
| 결제대기 주문 완료 처리 차단 | API | High |
| 취소 주문 상태 변경 차단 | API | High |
| 관리자 상태 변경 후 사용자 화면 반영 | E2E | High |

## 5.7 정산 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 완료되지 않은 주문이 정산에 반영 | 매출 데이터 오류 |
| 완료 주문이 정산에 미반영 | 매출 누락 |
| 취소/환불 주문이 정산에 포함 | 정산 신뢰도 저하 |
| 매장 A 정산에 매장 B 주문 포함 | 매장별 정산 오류 |
| 쿠폰 할인 전 금액이 정산에 반영 | 매출 금액 오류 |

### 분석

정산은 매장 운영 플랫폼에서 가장 중요한 데이터 검증 영역 중 하나이다.

정산 오류는 내부 운영 지표, 매출 관리, 가맹점 또는 매장별 평가 지표에 영향을 줄 수 있으므로 High 리스크로 분류한다.

본 포트폴리오에서는 완료 주문만 정산에 반영되는지를 핵심 기준으로 둔다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 완료 주문만 일 매출 반영 | API | High |
| 결제 실패 주문 정산 제외 | API | High |
| 취소완료 주문 정산 제외 | API | High |
| 환불완료 주문 정산 제외 | API | High |
| 매장별 정산 데이터 분리 | API | High |
| 쿠폰 할인 적용 후 금액 정산 | API | Medium |

## 5.8 권한 영역

### 주요 리스크

| 리스크 | 영향 |
|---|---|
| 일반 사용자가 관리자 페이지 접근 | 주문/매출 정보 노출 |
| 일반 사용자가 관리자 API 호출 | 보안 취약점 |
| 매장 A 관리자가 매장 B 주문 조회 | 매장 데이터 유출 |
| 매장 직원이 정산 정보 조회 | 권한 정책 위반 |
| 권한 변경 후 기존 세션에 반영되지 않음 | 접근 제어 오류 |

### 분석

권한 검증은 화면 접근과 API 접근을 모두 검증해야 한다.

화면에서 메뉴가 숨겨져 있더라도 API 직접 호출이 가능하면 심각한 보안 결함으로 판단한다.

특히 매장 간 데이터 접근 제어는 운영 플랫폼에서 매우 중요한 검증 포인트이다.

### 대응 전략

| 대응 테스트 | 테스트 유형 | 우선순위 |
|---|---|---|
| 일반 사용자 관리자 페이지 접근 차단 | E2E | High |
| 일반 사용자 관리자 API 호출 차단 | API | High |
| 매장 A 관리자의 매장 B 주문 조회 차단 | API | High |
| 매장 A 관리자의 매장 B 정산 조회 차단 | API | High |
| 본사 관리자의 전체 매장 정산 조회 가능 | API | Medium |
| 권한 없는 메뉴 미노출 | E2E/Manual | Medium |

## 6. 우선 검증 대상

본 프로젝트에서 우선 검증해야 할 대상은 다음과 같다.

| 우선순위 | 검증 대상 | 이유 |
|---|---|---|
| 1 | 결제 실패 처리 | 금전 데이터와 직접 연결 |
| 2 | 중복 결제/중복 주문 방지 | 사용자 피해 및 운영 혼선 방지 |
| 3 | 주문 취소 후 쿠폰/포인트 복원 | 사용자 혜택 데이터 정합성 |
| 4 | 완료 주문만 정산 반영 | 매출 데이터 신뢰성 |
| 5 | 매장 간 권한 분리 | 운영 데이터 보호 |
| 6 | 사용자 주문 후 관리자 화면 반영 | 매장 주문 처리 핵심 흐름 |
| 7 | 관리자 상태 변경 후 사용자 화면 반영 | 화면 간 데이터 동기화 |
| 8 | 좌석 중복 선택 차단 | 매장 운영 혼선 방지 |

| 자동화 우선순위 | 테스트 대상 | 테스트 유형 | 관련 리스크 |

| 1 | 결제 실패 시 주문 상태 결제대기 유지 | API | RISK-002 |
| 2 | 결제 실패 주문 정산 미반영 | API | RISK-003 |
| 3 | 주문 취소 시 쿠폰/포인트 복원 | API | RISK-004, RISK-005 |

| 5 | 일반 사용자의 관리자 API 접근 차단 | API | RISK-008 |

## 8. 리스크 분석 결론

본 프로젝트에서 가장 중요한 품질 리스크는 결제, 정산, 쿠폰/포인트, 권한, 주문 상태 변경 영역에 집중되어 있다.

따라서 테스트 설계와 자동화 대상 선정 시 다음 기준을 우선 적용한다.

1. 금전 데이터와 연결되는 결제/정산 영역은 High 우선순위로 검증한다.
2. 사용자 혜택 데이터와 연결되는 쿠폰/포인트 복원은 High 우선순위로 검증한다.
3. 운영 데이터 보호와 연결되는 권한 검증은 화면과 API 모두 검증한다.
4. 사용자 주문과 관리자 주문 처리가 연결되는 흐름은 E2E 테스트로 검증한다.
5. 상태 변경, 정산 반영, 권한 차단처럼 결과가 명확한 영역은 API 테스트로 우선 검증한다.
