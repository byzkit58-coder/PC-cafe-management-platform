# AI 테스트케이스 생성 프롬프트

## 1. 문서 목적

본 문서는 PC방 매장 운영 플랫폼의 요구사항을 기반으로 AI 도구를 활용해 테스트케이스 초안을 생성하기 위한 프롬프트를 정의한다.

AI는 테스트케이스 초안 생성 보조 도구로 사용하며, 최종 테스트케이스 확정은 QA 리뷰를 통해 수행한다.

---

## 2. 프롬프트

너는 10년 이상 경력의 QA 리드다.

아래 요구사항을 기반으로 PC방 매장 운영 플랫폼의 테스트케이스 초안을 생성해줘.

## 서비스 개요

PC방 매장 운영 플랫폼은 일반 사용자, 매장 직원, 매장 관리자, 본사 관리자가 사용하는 Web 기반 운영 관리 서비스이다.

- 일반 사용자는 로그인 후 좌석을 선택하고 음식 주문 및 결제를 진행한다.
- 매장 직원은 주문 접수, 조리중, 완료 상태로 주문을 처리한다.
- 매장 관리자는 담당 매장의 주문, 좌석, 매출, 정산을 확인한다.
- 본사 관리자는 전체 매장 정산과 쿠폰 정책을 관리한다.

## 핵심 기능

- 회원/로그인
- 좌석 선택
- 음식 주문
- 결제 성공/실패
- 쿠폰 사용/복원
- 포인트 사용/복원
- 주문 상태 변경
- 정산 반영
- 권한별 접근 제어
- 매장 간 데이터 접근 제한

## 중점 QA 관점

다음 관점을 반드시 포함해줘.

1. 정상 케이스
2. 예외 케이스
3. 권한별 접근 제어
4. 매장 간 데이터 분리
5. 데이터 정합성
6. 상태 변경
7. 결제 실패 처리
8. 주문 취소 후 쿠폰/포인트 복원
9. 완료 주문만 정산 반영
10. 중복 주문/중복 결제 방지
11. 사용자 화면과 관리자 화면 간 데이터 동기화
12. 자동화 적합도 판단

## 요구사항 요약

1. 사용자는 유효한 계정으로 로그인할 수 있다.
2. 비로그인 사용자는 좌석, 주문, 결제 화면에 접근할 수 없다.
3. 사용자는 사용 가능한 좌석을 선택할 수 있다.
4. 사용 중인 좌석은 다른 사용자가 선택할 수 없다.
5. 좌석을 사용 중인 사용자만 음식 주문을 생성할 수 있다.
6. 품절 메뉴는 주문할 수 없다.
7. 주문 버튼을 여러 번 클릭해도 주문은 1건만 생성되어야 한다.
8. 결제 성공 시 주문 상태는 결제완료로 변경되어야 한다.
9. 결제 실패 시 주문 상태는 결제대기로 유지되어야 한다.
10. 결제 실패 주문은 정산에 반영되지 않아야 한다.
11. 결제 버튼을 여러 번 클릭해도 결제 데이터는 1건만 생성되어야 한다.
12. 주문 취소 시 사용한 쿠폰은 복원되어야 한다.
13. 주문 취소 시 사용한 포인트는 복원되어야 한다.
14. 완료 상태의 주문만 일 매출에 반영되어야 한다.
15. 취소완료, 결제실패, 환불완료 주문은 정산에 반영되지 않아야 한다.
16. 일반 사용자는 관리자 페이지에 접근할 수 없다.
17. 일반 사용자는 관리자 API를 호출할 수 없다.
18. 매장 A 관리자는 매장 B 주문과 정산 데이터를 조회할 수 없다.
19. 본사 관리자는 전체 매장 정산 데이터를 조회할 수 있다.
20. 권한이 없는 메뉴는 화면에 노출되지 않아야 한다.

## 우선순위 기준

- High: 결제, 정산, 권한, 데이터 정합성, 중복 결제, 사용자 피해와 직접 관련
- Medium: 사용자 불편, 운영 혼선, 상태 표시 오류와 관련
- Low: 문구, 정렬, 낮은 영향도의 사용성 문제

## 자동화 적합도 기준

- E2E: 사용자 화면과 관리자 화면이 연결되는 실제 흐름 검증
- API: 상태 변경, 권한, 정산, 쿠폰/포인트처럼 결과가 명확한 검증
- Manual: UI/UX, 문구, 시각적 판단이 필요한 검증

## 추가 요청

- 단순 정상 케이스만 만들지 말고 예외 케이스를 충분히 포함해줘.
- 결제/정산/권한 관련 케이스는 High 우선순위로 분류해줘.
- 자동화 적합도는 E2E, API, Manual 중 하나 또는 복수로 표시해줘.
- 경력직 QA 포트폴리오에 넣을 수 있는 수준으로 구체적으로 작성해줘.
